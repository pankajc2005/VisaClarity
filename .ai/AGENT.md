## 5. Hard Rules — The Top 10 Mistakes Agents Make

### 5.1 Server logic = `createServerFn`, not edge functions

- File goes in `src/lib/<feature>.functions.ts` (client-safe path).
- Read `process.env.*` **only inside `.handler()`** — never at module scope.
- Use `.inputValidator()` with Zod before `.handler()`.
- Server-only helpers live in `*.server.ts` siblings.

### 5.2 Use `requireSupabaseAuth` for user data

- Any fn that reads/writes user rows MUST have `.middleware([requireSupabaseAuth])`.
- Components call via `useServerFn` + `useQuery`.
- Loaders for protected fns are **only safe under `src/routes/_authenticated/*`** — anywhere else, SSR/prerender has no session and the build fails with `Unauthorized`.

### 5.3 No client import of `*.server.ts`

The template's import protection blocks `*.server.ts` from client bundles by filename. If you need shared logic, expose it via a `.functions.ts` wrapper.

### 5.4 Tables in `public` schema MUST have GRANT statements

PostgREST does NOT grant default privileges. RLS alone is not enough. Order inside every migration:

```sql
CREATE TABLE public.x (...);
GRANT SELECT, INSERT, UPDATE, DELETE ON public.x TO authenticated;
GRANT ALL ON public.x TO service_role;
-- GRANT SELECT ON public.x TO anon;  -- ONLY if a policy allows anon reads
ALTER TABLE public.x ENABLE ROW LEVEL SECURITY;
CREATE POLICY ...;
```

### 5.5 User roles live in a separate table

Never store `role` on profiles/users. Use `public.user_roles` + `has_role(uuid, app_role)` security-definer function. Pattern is already in the DB — see `user_roles` table.

### 5.6 Design tokens only — never hardcoded colors

No `text-white`, `bg-black`, `text-[#fff]` in components. Use semantic tokens defined in `src/styles.css` (`--background`, `--foreground`, `--primary`, etc.) in `oklch`. Add new tokens there before using them.

### 5.7 No Node-only packages

Workers runtime (`workerd`) does **NOT** support: `sharp`, `puppeteer`, `canvas`, `child_process`, `fs.watch`, native bindings, `node-gyp` packages. Symptoms: `[unenv] X is not implemented yet!`, `__dirname is not defined`, works-in-dev-fails-in-prod. Find a Worker-compatible alternative or call an external HTTP API.

### 5.8 Route conventions

- Flat dot-separated naming: `blog.$slug.tsx`, NOT `blog/$slug.tsx`
- No trailing slashes in paths
- No `src/pages/` — TanStack uses `src/routes/`
- Every route with a loader needs `errorComponent` + `notFoundComponent`
- Root route needs `notFoundComponent`
- Parent layout routes MUST render `<Outlet />`

### 5.9 Import boundaries

- Router APIs (`createFileRoute`, `Link`, `useRouter`, `Outlet`): from `@tanstack/react-router`
- Server fn APIs (`createServerFn`, `useServerFn`): from `@tanstack/react-start`
- Never mix these. `createFileRoute` is NOT in `@tanstack/react-start`.

### 5.10 Free beta — no paywalls in UI

v1 ships 100% free. Pro / Pro Max code stays behind flags in `src/lib/platform.ts`. Do not surface `LockedFeature`, upgrade modals, or pricing tiers without explicit user approval. See `LAUNCH.md`.

## 6. AI Calls — The Only Pattern

All chat completions go through **one** seam:

```ts
// src/lib/ai-gateway.server.ts
import { chatCompletion } from "@/lib/ai-gateway.server";

const { choices } = await chatCompletion({
  model: "google/gemini-2.5-flash-lite",   // cheapest capable — default
  temperature: 0.25,
  max_tokens: 4096,
  messages: [
    { role: "system", content: SYSTEM_PROMPT },
    { role: "user", content: userInput },
  ],
  response_format: { type: "json_schema", json_schema: { ... } },
});
```

Rules:

- Never call OpenAI/Google/Anthropic SDKs directly. The gateway is portable (env-overridable to OpenRouter, Together, Groq, etc.).
- Default model: `google/gemini-2.5-flash-lite`. Upgrade per task:
  - Heavy reasoning / multimodal → `google/gemini-2.5-pro` or `openai/gpt-5`
  - Image generation → `google/gemini-2.5-flash-image` (Nano Banana)
- Always request structured output via `response_format` JSON schema when the result feeds UI.
- Never fabricate URLs in `sources[]` — only use URLs returned by the research agent (Tavily).

## 7. Database Workflow

- Schema changes → use the migration tool (handles user approval).
- Never `ALTER DATABASE postgres` — not allowed.
- Never touch reserved schemas: `auth`, `storage`, `realtime`, `supabase_functions`, `vault`.
- Use **validation triggers**, not `CHECK` constraints with `now()` (CHECK must be immutable).
- After `CREATE TABLE`: **GRANT → ENABLE RLS → CREATE POLICY** (in that order).
- Default to user-scoped policies: `auth.uid() = user_id`.
- For roles: use `public.has_role(auth.uid(), 'admin')` in policies — never query `user_roles` directly in a policy (recursion).
- Don't add foreign keys to `auth.users` — store `user_id uuid` and rely on RLS.

Existing tables (snapshot — see `src/integrations/supabase/types.ts` for source of truth):
`error_reports`, `idempotency_keys`, `leads`, `personalized_requests`, `personalized_roadmap_requests`, `roadmap_cache`, `roadmap_usage`, `saga_steps`, `saved_roadmaps`, `user_roles`, `user_subscriptions`.

## 8. Auth Rules

- Email + Google sign-in only. Call `supabase--configure_social_auth` with `providers: ["google"]` whenever you add Google sign-in.
- **Never** enable anonymous signups.
- **Never** enable auto-confirm email unless the user explicitly asks.
- Admin surfaces gated via `user_roles` + founder email check — never via `localStorage` or hardcoded credentials.

## 10. Trust Rules (Product-Level, Not Just Code)

VisaClarity is a **trust product**. Breaking these is worse than breaking the build.

- **Never** write copy claiming "guaranteed", "100% approval", "official partner", "we book your slot".
- Every corridor output must ship with a **"Last verified: <date>"** stamp.
- `sources[]` arrays must use URLs returned by Tavily verbatim — never fabricate, never guess.
- Empty / unverified corridors must not ship — hide them rather than show stale data.
- Disclaimers: link `/legal/disclaimer` from every roadmap output.
- See `LAUNCH.md` §"Things We Will NEVER Ship" for the permanent ban list (auto-booking bots, fake urgency timers, selling user data, etc.).

## 12. Pre-Commit Self-Check

Before declaring a task done, the agent mentally runs:

- [ ] Did I edit any file in §4 (auto-generated)? → Revert.
- [ ] Did I import a `*.server.ts` from client code? → Move logic into a `*.functions.ts` wrapper.
- [ ] Did I add a `public` table without GRANTs? → Add them in the same migration.
- [ ] Did I hardcode a color or font? → Replace with a token from `src/styles.css`.
- [ ] Did I add a Node-only npm package? → Find a Worker-compatible alternative.
- [ ] Did I expose a paid feature in UI? → Gate it behind `src/lib/platform.ts`.
- [ ] Did I add unverifiable marketing copy ("guaranteed", "100%")? → Remove it.
- [ ] Did I create a route file but use `<Link>` to a path that doesn't exist yet? → Create the route file first.
- [ ] Did I read `process.env.*` at module scope of a `*.functions.ts`? → Move inside `.handler()`.
- [ ] Did I forget `errorComponent` / `notFoundComponent` on a new route? → Add them.

---

## 13. Verification Habits

- After code edits: check build output / typecheck signal (do not run builds manually — the harness runs them).
- After schema changes: re-read your SQL and confirm GRANT block is present.
- After AI prompt changes: run one sample through the affected fn and inspect the output before claiming it works.
- After UI changes: respect the user's current viewport — VisaClarity must look good on mobile (846px and narrower).

---

## 14. When in Doubt

**Ask the user.** Better to clarify than to ship a wrong assumption. This codebase is a trust product — silent guessing breaks the brand.

If the user gives a vague request ("make it better", "fix it"), use `questions--ask_questions` (when available) to clarify before editing.

## Blog v2 (Dec 2026)

Blog is **DB-backed**. Static file `src/lib/blog/posts.ts` is legacy — referenced only by roadmap pages for related-guide suggestions and by the one-shot importer.

**Tables:** `blog_authors`, `blog_posts` (status enum: draft/in_review/published/archived), `blog_topic_queue`, `blog_generation_logs`. RLS: public reads only `published`. Admin manages all.

**Storage:** Private `blog-images` bucket with public-read RLS policy on `storage.objects`.

**Server libs:**

- `src/lib/blog/research-agent.server.ts` — Tavily, Wikipedia, Wikidata, Crossref, OpenAlex, HN, GDELT, DuckDuckGo, direct .gov fetch, Tinyfish (optional).
- `src/lib/blog/image-gen.server.ts` — Lovable Gateway gpt-image-2 → gemini-3.1-flash-image → Pollinations (no card needed).
- `src/lib/blog/generate-post.server.ts` — orchestrator (research → gemini-2.5-pro draft → image → save).
- `src/lib/blog/writer-system.md` — anti-hallucination system prompt. Refuses if insufficient official sources.
- `src/lib/blog/storage.server.ts` — upload to bucket via supabaseAdmin.

**Server fns** in `src/lib/blog.functions.ts` (public reads + admin CRUD + queue mgmt) and `src/lib/blog/import-legacy.functions.ts` (one-shot static import).

**Public API:** `POST /api/public/blog/generate` — `BLOG_API_KEY` bearer auth. Body: `{ topic, primary_keyword, secondary_keywords?, audience?, angle?, author_slug?, official_seeds?, auto_publish?, idempotency_key? }`. `auto_publish: true` is honored only when confidence=high and zero `[NEEDS OFFICIAL SOURCE]` markers.

**Admin UI:** `/admin/blog` (admin role required). Tabs: Queue / Posts. Editor: live edit + status flip + hero regen + sources panel + generation log.

**Reader UX:** `blog.index.tsx` (DB-backed, hero + grid), `blog.$slug.tsx` (Medium-style serif column, progress bar, TL;DR, MDX-lite renderer, sources, author byline → `blog.author.$slug.tsx`).

**Secrets:** `BLOG_API_KEY` (required for public API), `TINYFISH_API_KEY` (optional). `LOVABLE_API_KEY` + `TAVILY_API_KEY` already present.

**Naming rule:** Author byline shows `name · locale_hint` only. Never country, never flag.
